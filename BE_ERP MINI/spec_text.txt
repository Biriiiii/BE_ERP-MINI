MODULE NHÂN SỰ (HR) 3.1 Hồ sơ nhân viên Business logic Mã NV tự sinh: NV-[YYYY]-[SEQ 3 số]. Không cho sửa tay. Hợp đồng sắp hết hạn ≤ 30 ngày: gửi thông báo tự động cho STORE_MANAGER mỗi sáng 8h. Nhân viên nghỉ việc: status = TERMINATED, vô hiệu hoá tài khoản ngay. Không xóa hồ sơ. Thay đổi lương: tạo bản ghi SalaryHistory (ngày hiệu lực, lương cũ, lương mới, lý do). Không overwrite. 3.2 Chấm công Chấm công QR code cá nhân tại tablet đặt cửa vào. Ghi timestamp thực tế check-in / check-out. Đến trễ &gt; 10 phút: ghi LATE + số phút trễ. Grace period ≤ 10 phút: không phạt. OT phải được STORE_MANAGER pre-approve. OT không có duyệt trước → không tính tiền. Quên check-out lúc 23:59 → đánh dấu MISSING_CHECKOUT, STORE_MANAGER điều chỉnh ngày hôm sau. 3.3 Nghỉ phép Loại nghỉ Số ngày/năm Hưởng lương? Ghi chú Nghỉ phép năm 12 ngày ✅ Có Prorated nếu chưa đủ 1 năm Nghỉ bệnh có phép Không giới hạn (có giấy) ✅ BHXH trả Cần nộp giấy khám Nghỉ việc riêng 3 ngày/năm ✅ Có Kết hôn, tang... Nghỉ không phép — ❌ Không Trừ lương, ảnh hưởng KPI 3.4 Tính lương Lương gross = Lương CB + Phụ cấp + OT - Trừ nghỉ KP - Trừ đi trễ Hệ số OT: ngày thường 1.5x | ngày nghỉ 2.0x | lễ tết 3.0x BHXH 8% + BHYT 1.5% + BHTN 1% = 10.5% gross Thuế TNCN: biểu lũy tiến 7 bậc theo TT111/2013 Lương NET = Gross - BHXH - BHYT - BHTN - Thuế TNCN ⚠️ Lock lương: Bảng lương sau khi OWNER duyệt → LOCK hoàn toàn. MODULE HÀNG HÓA &amp; KHO (INVENTORY) 4.1 Phân nhóm hàng Nhóm Ví dụ Đơn vị Quản lý đặc biệt Rau củ quả Cải, cà chua, khoai kg, bó, túi HSD ≤ 3 ngày, cân trọng lượng Thịt tươi sống Heo, gà, bò kg, miếng HSD ≤ 2 ngày, kho lạnh 0–4°C Thủy hải sản Cá, tôm, mực kg, con HSD ≤ 1 ngày, kho lạnh Thực phẩm chế biến Giò chả, nem chua hộp, gói HSD 3–14 ngày Sữa &amp; đồ uống Sữa tươi, nước ngọt hộp, chai, lốc HSD tuần–tháng Hàng khô / FMCG Gạo, dầu, mì gói gói, thùng HSD tháng–năm Đồ gia dụng Nồi, chén, khăn cái, bộ Không có HSD 4.2 Business logic danh mục Mã SKU tự sinh: [NHOM]-[SEQ], ví dụ RAU-0021, FMCG-0145. Đơn vị tính kép: đơn vị nhập (thùng 24 chai) ↔ đơn vị bán (chai). Hệ thống tự quy đổi. Giá vốn: Weighted Average Cost — tính lại sau mỗi lần nhập kho. min_stock_level: khi tồn ≤ min → tự động tạo ReplenishmentAlert cho WAREHOUSE_STAFF. 4.3 Nhập kho Tươi sống (nhập hàng ngày, không có PO trước) Hàng về sáng sớm → WAREHOUSE_STAFF cân/đếm thực tế → Nhập: sản phẩm + số lượng + đơn giá hôm nay + HSD → tạo Lot mới → STORE_MANAGER xác nhận → Cập nhật StockBalance → Tạo công nợ AP → Kế toán ghi nhận FMCG / hàng khô (có PO từ trước) WAREHOUSE_STAFF tạo PR → STORE_MANAGER duyệt → Sinh PO draft → OWNER duyệt PO (nếu &gt; ngưỡng giá trị) → Hàng về → scan barcode đối chiếu PO → 3-way match: PO ↔ Phiếu nhập ↔ Hóa đơn NCC → Tạo AP → Kế toán 4.4 Xuất kho Loại xuất Trigger Ai tạo Ai duyệt Journal Entry Xuất bán (POS) Giao dịch POS Hệ thống tự động Không cần Nợ 632 / Có 156 Xuất trả NCC Hàng lỗi WAREHOUSE_STAFF STORE_MANAGER Nợ 331 / Có 156 Xuất nội bộ Dùng cho ST WAREHOUSE_STAFF STORE_MANAGER Nợ 642 / Có 156 Hao hụt / Hủy Hàng hỏng, hết HSD WAREHOUSE_STAFF STORE_MANAGER Nợ Hao hụt / Có 156 FEFO bắt buộc: lô có HSD gần nhất xuất trước. YÊU CẦU PHI CHỨC NĂNG 6.1 Performance POS giao dịch: response ≤ 1 giây (scan → giá → in bill). Trang tồn kho ~3.500 SKU: load ≤ 2 giây, pagination 50 dòng/trang. Báo cáo doanh thu tháng: generate ≤ 5 giây. POS offline-first: bán hàng, lưu local khi mất mạng, sync khi có kết nối. 6.2 Bảo mật Mật khẩu: bcrypt, tối thiểu 8 ký tự. JWT expire 8h. POS terminal: PIN 4–6 số + device binding. Sensitive data (lương, CCCD, tài khoản NH): AES-256. Backup tự động lúc 2h sáng → Google Drive / S3. Rate limiting: 60 req/phút/IP. 6.3 Usability POS: giao diện cảm ứng, nhân viên học dùng &lt; 30 phút. Thông báo lỗi tiếng Việt rõ ràng, không dùng mã lỗi kỹ thuật. Mobile-friendly: OWNER và STORE_MANAGER xem báo cáo trên điện thoại được. Hỗ trợ máy in bill nhiệt 80mm phổ biến: Xprinter, Epson TM. 7. Mọi thay đổi dữ liệu đều phải truy vết được: ai làm, làm gì, lúc nào, thay đổi từ giá trị nào sang giá trị nào. 10.1 Các trường bắt buộc trên MỌI bảng nghiệp vụ Mỗi bảng trong database phải có đủ các cột sau — dev KHÔNG được bỏ qua: Tên cột Kiểu dữ liệu Nullable Mô tả id UUID / BIGINT NOT NULL Khóa chính, tự sinh created_at TIMESTAMP WITH TZ NOT NULL Thời điểm tạo bản ghi (UTC) created_by UUID → FK users.id NOT NULL ID người tạo updated_at TIMESTAMP WITH TZ NULL Thời điểm cập nhật gần nhất updated_by UUID → FK users.id NULL ID người cập nhật gần nhất deleted_at TIMESTAMP WITH TZ NULL NULL = chưa xóa; có giá trị = đã xóa mềm deleted_by UUID → FK users.id NULL ID người thực hiện xóa mềm is_deleted BOOLEAN NOT NULL DEFAULT false. Index cột này. version INTEGER NOT NULL Optimistic locking — tăng 1 sau mỗi lần UPDATE ⚠️ Quan trọng: created_by và created_at KHÔNG được phép UPDATE sau khi đã insert. Backend phải ignore nếu client gửi lên. Chỉ set một lần duy nhất lúc INSERT. 🔒 Soft delete bắt buộc: Không dùng DELETE trực tiếp cho bất kỳ bảng nghiệp vụ nào. Chỉ set deleted_at = NOW(), deleted_by = current_user_id, is_deleted = true. Mọi query phải thêm điều kiện WHERE is_deleted = false. 10.2 Bảng audit_log — Cấu trúc chi tiết Ngoài các cột trên từng bảng, hệ thống duy trì một bảng audit_log trung tâm, ghi lại TOÀN BỘ thay đổi của mọi entity. Không được viết tay từng chỗ — dễ bị bỏ sót. // audit.service.ts (TypeScript example) async function logAudit(params: { action: AuditAction; // &apos;CREATE&apos; | &apos;UPDATE&apos; | ... entityType: string; // &apos;Employee&apos;, &apos;Product&apos;, ... entityId: string; entityLabel?: string; // Tên hiển thị oldValue?: Record&lt;string, any&gt;; // Snapshot trước khi thay đổi newValue?: Record&lt;string, any&gt;; // Snapshot sau khi thay đổi reason?: string; // Lý do (bắt buộc với một số action) context: RequestContext; // user, ip, device, session }) { const changedFields = computeDiff(params.oldValue, params.newValue); await db.auditLog.insert({ id: uuid(), timestamp: new Date(), user_id: context.user?.id ?? null, user_name: context.user?.name ?? &apos;SYSTEM&apos;, user_role: context.user?.role ?? &apos;SYSTEM&apos;, ip_address: context.ip, device_info: context.deviceInfo, session_id: context.sessionId, action: params.action, entity_type: params.entityType, entity_id: String(params.entityId), entity_label: params.entityLabel, old_value: maskSensitive(params.oldValue), new_value: maskSensitive(params.newValue), changed_fields: changedFields, reason: params.reason, source: context.source ?? &apos;USER&apos;, }); } ⚠️ Transaction: Việc ghi audit log phải nằm trong cùng DB transaction với thao tác chính. Nếu thao tác chính rollback → audit log cũng rollback. Không được ghi audit log sau khi commit xong (có thể mất log nếu server crash). 10.5.3 Tính diff changed_fields // Tính các field thực sự thay đổi (không log những field không đổi) function computeDiff( oldVal: Record&lt;string, any&gt;, newVal: Record&lt;string, any&gt; ): Record&lt;string, { from: any; to: any }&gt; { const diff: Record&lt;string, { from: any; to: any }&gt; = {}; const allKeys = new Set([...Object.keys(oldVal ?? {}), ...Object.keys(newVal ?? {})]); for (const key of allKeys) { // Bỏ qua các audit fields — không diff chính mình if ([&apos;updated_at&apos;,&apos;updated_by&apos;,&apos;version&apos;].includes(key)) continue; if (JSON.stringify(oldVal?.[key]) !== JSON.stringify(newVal?.[key])) { diff[key] = { from: oldVal?.[key], to: newVal?.[key] }; } } return diff; } Ví dụ changed_fields khi cập nhật lương nhân viên: { &quot;base_salary&quot;: { &quot;from&quot;: 7000000, &quot;to&quot;: 8000000 }, &quot;meal_allowance&quot;: { &quot;from&quot;: 500000, &quot;to&quot;: 600000 } } 10.5.4 Optimistic Locking — tránh ghi đè nhau Khi 2 người cùng sửa một bản ghi, người sau phải báo lỗi thay vì ghi đè người trước: -- SQL UPDATE có kiểm tra version UPDATE employees SET base_salary = 8000000, updated_by = &apos;user-uuid-xyz&apos;, updated_at = NOW(), version = version + 1 WHERE id = &apos;emp-uuid-abc&apos; AND version = 5; -- version phía client đang giữ -- Nếu affected rows = 0 → bản ghi đã bị người khác sửa -- → Throw: OptimisticLockException(&apos;Dữ liệu đã bị thay đổi bởi người khác. -- Vui lòng tải lại trang và thử lại.&apos;) 10.6 API truy vấn lịch sử thay đổi API Endpoint Mô tả Quyền truy cập GET /audit/entity/{type}/{id} Xem toàn bộ lịch sử của 1 bản ghi cụ thể STORE_MGR (kho/HR), ACCOUNTANT (kế toán), OWNER GET /audit/user/{user_id} Xem mọi thao tác của 1 nhân viên OWNER, ACCOUNTANT GET /audit/feed Feed real-time các thay đổi (filter theo module) OWNER, ACCOUNTANT GET /audit/entity/{type}/{id}/diff/{v1}/{v2} So sánh 2 phiên bản cụ thể (version v1 và v2) OWNER, ACCOUNTANT GET /audit/export Export audit log ra CSV/Excel (có filter) OWNER, ACCOUNTANT Query parameters chuẩn cho /audit/entity và /audit/feed: GET /audit/entity/Employee/emp-001?
